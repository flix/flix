# The checkcast Invariant in Bytecode Generation

This document explains how the Flix compiler manages JVM type casts in
generated bytecode, why they are needed, and the design principles
behind the current implementation.

## Background: Type Erasure and GETFIELD

Flix compiles enums, tuples, records, and structs to JVM classes with
erased field types. For example, a `RedBlackTree.Node(Color, left, key,
value, right)` compiles to a class `Tag$Obj$Obj$Obj$Obj$Obj` where all
five fields are declared as `java.lang.Object`:

```java
public final class Tag$Obj$Obj$Obj$Obj$Obj extends Tagged$ {
    public Object v0;  // Color
    public Object v1;  // left subtree
    public Object v2;  // key
    public Object v3;  // value
    public Object v4;  // right subtree
}
```

When `GETFIELD` extracts a field, the JVM verifier sees the declared
type (`Object`), not the runtime type (`Tagged$`). To use the value as
its actual type, a `CHECKCAST` instruction must narrow the verifier
type.

## The Invariant

**`compileExpr(e)` leaves a value on the JVM stack whose verifier type
is compatible with `BackendType.toBackendType(e.tpe)`.**

Each expression is responsible for producing the correct type. `Let`
does not cast (with one exception noted below).

## Who Casts Where

### Operations that extract from erased storage

Six operations leave `Object` on the stack for reference types because
they read from erased fields or arrays. Each emits its own
`castIfNotPrim` after the extraction:

| Operation | Extraction | Cast target |
|-----------|-----------|-------------|
| `Untag` | `GETFIELD` on `Tag$...` | `BackendType.toBackendType(tpe)` |
| `Index` | `GETFIELD` on `Tuple$...` | `BackendType.toBackendType(tpe)` |
| `RecordSelect` | `GETFIELD` on `RecordExtend` | `BackendType.toBackendType(tpe)` |
| `StructGet` | `GETFIELD` on `Struct$...` | `BackendType.toBackendType(tpe)` |
| `ArrayLoad` | `AALOAD` | `BackendType.toBackendType(tpe)` |
| `Unbox` | `GETFIELD` on `Value$.o` | `BackendType.toBackendType(tpe)` |

### Function calls

Function calls (`ApplyClo`, `ApplyDef`, `RunWith`) are wrapped by the
Eraser in `Unbox(Cast(call, type))`. The `Cast` node emits
`castIfNotPrim` to convert the unboxed `Object` result to the declared
type. No additional cast is needed.

### Other expressions

- **`Tag` construction**: Produces a specific `Tag$...` subclass of
  `Tagged$`. The JVM verifier accepts this as a `Tagged$` (subtype
  relationship). No cast needed.
- **`Var`**: Loads from a local variable slot. The slot has whatever
  type was stored by the enclosing `Let`. Since Flix has no subtyping,
  all Var references to the same symbol have the same type as the Let
  that defined it (verified by AST consistency checks).
- **`IfThenElse` / `Branch`**: The JVM verifier computes the
  least-upper-bound (LUB) of the branch types at the join point.
  For Flix enums, both branches produce `Tagged$` subtypes, so
  the LUB is `Tagged$` -- which matches the declared type.
- **Constants, closures, `Is` checks, etc.**: Already produce the
  correct type.

### `Let` bindings

`Let` does **not** cast, with one exception: `NewObject` expressions
(anonymous Java class construction). The JVM verifier cannot resolve
the generated anonymous class name at merge points and falls back to
`Object`. The cast forces the verifier to see the declared superclass
type. All other expression types produce the correct JVM stack type
without Let's help.

### `NewObject` -- the remaining exception

When Flix creates an anonymous Java class via `new SomeJavaClass { ... }`,
the generated class has a synthetic name (e.g., `Anon$42`) that extends
`SomeJavaClass`. The JVM verifier resolves class names to check subtype
relationships at merge points. Because the anonymous class is generated
in memory by Flix's classloader, the verifier may not be able to resolve
the class hierarchy and falls back to `Object`.

The `Let` cast for `NewObject` narrows the verifier type from the
anonymous class to the declared superclass (e.g., `SomeJavaClass`),
which the verifier can resolve. This is a workaround for a JVM verifier
limitation with dynamically generated classes.

## Effect System: Frames and Suspension

### How effects use frames

When a function uses algebraic effects, it may need to suspend (save
its state) and resume later. The compiler generates the function body
inside an `applyFrame` method with this structure:

```
class Clo$example extends Fn1$Obj$Obj implements Frame {
    // Saved locals (erased types for reference fields)
    public Object l0;    // e.g., a Tagged$ at runtime
    public int    l1;    // primitives keep their exact type
    public Object l2;    // e.g., a String at runtime
    // Closure captures
    public Tagged$ clo0;
    // Function arguments (erased)
    public Object arg0;
    // Resume address
    public int pc;

    Result applyFrame(Value resumptionArg) {
        // 1. RESTORE: load locals from fields, cast to real types
        Tagged$ slot4 = (Tagged$) this.l0;
        int     slot5 = this.l1;
        String  slot6 = (String) this.l2;

        EnterLabel:  // tail-call loop target

        // 2. Load closure params and function args
        Tagged$ slotN = (Tagged$) this.arg0;

        // 3. DISPATCH on pc
        switch (this.pc) {
            case 0 -> default -> body start
            case 1 -> pcPointLabel1  // resume after 1st suspension
        }

        // 4. BODY CODE
        ...
        result = f(arg);  // impure call -- may suspend

        // If f returns a Suspension:
        //   SAVE: copy this, set pc=1, save locals to fields
        //   return Suspension with our frame attached
        // If f returns a Value:
        //   continue normally

        // pcPointLabel1 is placed here
        pcPointLabel1:
            narrowLocals();   // re-cast locals to declared types
            v = resumptionArg;
        afterUnboxing:
            // continue with v
    }
}
```

### pcPoints and suspension

Each impure call in the body gets a **pcPoint** -- a numbered resume
address. When the call suspends:

1. **Save** (`setPc`): clone the closure (`this.copy()`), set
   `frame.pc = N`, save each local to its erased field
   (`frame.l0 = slot4`, etc.). Since lparam fields are erased
   (`Object`), no cast is needed on save.
2. **Return**: attach the frame to the `Suspension` and return it up
   the call stack.
3. **Resume**: the handler calls `frame.applyFrame(value)`. The
   `tableswitch` jumps to `pcPointLabelN`. `narrowLocals` re-casts
   each local to its declared type.

### narrowLocals at resume sites

At each `pcPointLabel`, the JVM verifier has merged the local types
from the entry path (set by `loadFromField` at the top of
`applyFrame`) with whatever types existed at that bytecode offset in
the body code. This merge can broaden types.

`narrowLocals` immediately after the label re-casts each non-primitive
local by loading, casting, and re-storing it:

```java
pcPointLabel1:
    slot4 = (Tagged$) slot4;   // aload + checkcast + astore
    slot6 = (String)  slot6;   // aload + checkcast + astore
    // (primitives like slot5 don't need narrowing)
    v = resumptionArg;
```

This restores the exact declared types so the resumed code operates
correctly.

### The lparam field convention

Local variable fields (`l0`, `l1`, ...) use erased types, matching
the `fparams` convention (`arg0`, `arg1`, ...):

| Parameter kind | Field type | Load cast | Store cast |
|---------------|-----------|-----------|------------|
| `fparams` (`arg0`, ...) | erased (`Object`) | yes (`castTo`) | no |
| `lparams` (`l0`, ...) | erased (`Object`) | yes (`castTo`) | no |
| `cparams` (`clo0`, ...) | exact type | no | no |

This means `setPc` can always store locals without casting (the field
type is `Object`, which accepts any reference), and `loadFromField`
casts from `Object` to the real type on restore.

## Summary of Cast Sites

| Site | When | Why |
|------|------|-----|
| `Untag`, `Index`, `RecordSelect`, `StructGet`, `ArrayLoad`, `Unbox` | Always | Extraction from erased storage |
| `Cast` (from Eraser, wrapping function calls) | Always | Narrowing unboxed `Object` to declared type |
| `Let` | Only for `NewObject` | JVM verifier can't resolve anonymous class hierarchy |
| `narrowLocals` | At each `pcPointLabel` | Re-narrow locals after tableswitch merge at resume |
| `loadFromField` | lparams and fparams | Cast from erased field to real type on restore |
| `setPc` | Never (fields are erased) | -- |

## Eraser's Role

The Eraser does NOT wrap `Untag`, `Index`, `RecordSelect`, or
`StructGet` in `Cast` nodes. These operations carry their real type
(`t = visitType(tpe)`) directly, and their GenExpression handlers emit
the cast.

The Eraser DOES wrap `ApplyClo`, `ApplyDef`, and `RunWith` in
`Unbox(Cast(...))` to extract and cast function call results.
